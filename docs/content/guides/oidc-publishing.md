# GitHub Actions OIDC publishing (secretless)

This guide explains how **GitHub Actions OIDC** publishing works in nrdocs and what (if any) configuration is required.

## Who this guide is for

- **Repo owners (docs authors)**: you want your repo to publish on push, without managing credentials.
- **Platform operators (admins)**: you run the Control Plane and decide which repos may publish.

## What “secretless” means

The default `publish-docs.yml` workflow generated by `nrdocs init` does **not** require:

- a per-repository GitHub Actions secret like `NRDOCS_PUBLISH_TOKEN`
- a per-repository GitHub Actions variable like `NRDOCS_PROJECT_ID`
- `gh` CLI access to configure repository secrets/variables

Instead, each workflow run obtains **short-lived publish credentials** from the Control Plane using a GitHub-issued OIDC token.

## Repo owner: what you must do (functional checklist)

If you onboarded via `nrdocs init --token ...`, the workflow is already correct. Your steps are:

1. **Commit the generated files** (including `.github/workflows/publish-docs.yml`)
2. **Push to the publish branch** (the branch configured in the workflow trigger)
3. Check GitHub Actions → the “Publish Docs to nrdocs” workflow should run

You do **not** need:

- an admin API key
- a GitHub PAT
- to create Actions secrets/variables for publishing

### If you are creating a workflow manually

Ensure your workflow has:

1. **OIDC permission** (required):

```yaml
permissions:
  contents: read
  id-token: write
```

2. **OIDC token request + exchange** (required):

- request a GitHub OIDC token with audience set to your Control Plane URL
- exchange it at `POST /oidc/publish-credentials` to get `{ project_id, repo_publish_token }`

If you don’t do this, publishing will fail because the Control Plane will not accept the admin API key on the publish endpoint.

### Optional repo configuration

If your docs are not in the default directory, set:

- **GitHub Actions variable** `NRDOCS_DOCS_DIR` = `docs` (or your docs folder)

This is not a credential.

## Operator: what you must do (functional checklist)

For OIDC publishing to work for a repository, you must ensure:

1. **Control Plane is deployed with OIDC exchange**
   - endpoint exists: `POST /oidc/publish-credentials`
   - outbound network works to fetch GitHub JWKS:
     - `https://token.actions.githubusercontent.com/.well-known/jwks`
2. **Project is approved**
3. **Project has correct `repo_identity`**
   - canonical: `github.com/<owner>/<repo>`
   - must match the repo that will run GitHub Actions

If any of these are missing, the exchange will fail (typically 401/404/409).

## How the auth flow works (high level)

On each publish run:

1. **GitHub Actions issues an OIDC token** to the workflow job (JWT), because the job has `permissions: id-token: write`.
2. The workflow requests a token with an explicit **audience** set to the Control Plane URL it will call.
3. The workflow calls the Control Plane exchange endpoint:

   - `POST /oidc/publish-credentials`
   - `Authorization: Bearer <github-oidc-jwt>`

4. The Control Plane:

   - verifies the JWT signature against GitHub’s public JWKS
   - verifies standard claims (`iss`, `aud`, `exp`)
   - reads `repository = "owner/repo"` from the token and maps it to repo identity `github.com/owner/repo`
   - looks up the project by `repo_identity`
   - ensures the project is **approved**
   - mints a short-lived **repo publish JWT** and returns it along with the `project_id`

5. The workflow publishes by calling:

   - `POST /projects/<project_id>/publish`
   - `Authorization: Bearer <repo-publish-jwt>`
   - `X-Repo-Identity: github.com/${{ github.repository }}`

## Control Plane requirements

Your deployed Control Plane must include the OIDC exchange endpoint:

- `POST /oidc/publish-credentials`

and it must be able to fetch GitHub’s OIDC JWKS URL at runtime:

- `https://token.actions.githubusercontent.com/.well-known/jwks`

## Project requirements

For OIDC exchange to succeed:

- the project must exist in D1
- the project must be **approved**
- the project must have a `repo_identity` that matches GitHub’s `owner/repo`:
  - expected canonical form: `github.com/<owner>/<repo>`

Bootstrap onboarding (`nrdocs init --token ...`) sets `repo_identity` for you (inferred from the git `origin` remote unless overridden).

Operator onboarding via the admin API must set the project `repo_identity` (or use the CLI `approve --repo-identity ...`) so that OIDC can map the repo to the correct project.

## GitHub Actions workflow requirements

The workflow must:

- request an id-token:

  - `permissions: id-token: write`

- request the OIDC token with audience = the Control Plane API base URL (origin)

The `nrdocs init` generated workflow already does this.

## Troubleshooting

### “OIDC exchange failed”

Typical causes:

- Control Plane is not updated/deployed with `POST /oidc/publish-credentials`
- project is not approved
- project has no `repo_identity`, or it does not match `github.com/<owner>/<repo>`

### “No project is registered for repository github.com/<owner>/<repo>”

The Control Plane cannot find a project with `repo_identity = github.com/<owner>/<repo>`.

Fix by:

- onboarding via `nrdocs init --token ...` from the correct repo, or
- ensuring the operator set `repo_identity` for the project.
