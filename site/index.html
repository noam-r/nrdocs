Open-source docs publishing

Keep docs with the code. Publish only the docs.

nrdocs is an open-source publishing tool for documentation stored in GitHub repositories. Manage docs alongside your code, publish them without running servers, and choose whether the generated site is public or protected.

View on GitHub See it in action 
# one-time on your machine
nrdocs config set api-url 'https://<control-plane>.workers.dev'

# in your Git repo
nrdocs init
git add . && git commit -m "Add docs" && git push

The point

Documentation gets worse when it lives somewhere else.

A separate docs repo sounds harmless until code changes, docs changes, reviews, releases, and ownership all start drifting apart. nrdocs lets docs stay in the project repo while publishing only the generated documentation site.

Docs live with code

Keep documentation in the same repository as the implementation, where pull requests, ownership, and release work already happen.

Visibility is separate

Use public or private repos. Publish public or protected docs. Those should be separate choices.

No servers to run

nrdocs runs on Cloudflare Workers, D1, and R2. No VM, no container, no tiny forgotten server.

Access control built in

Serve docs publicly, protect them with a password today, and support invite-only access by email in the future.

Workflow

A normal Git workflow for documentation.

Write docs where the work happens. Review them with code changes. Push as usual. The first CI run auto-registers the site with OIDC, an operator approves, and the same run publishes and serves the generated site without requiring a server or a separate public mirror repository.

1
Write docs with code Keep Markdown in the project repo, close to the implementation it documents.
2
Review and push Use the same GitHub review flow you already use for code changes.
3

4

5

Good for

Good for projects where docs should move with the work.

nrdocs is intentionally boring infrastructure. That is the idea. You update docs where the work is happening, push them, and get a URL.

  • Public docs for private products. Keep the source private and publish only the generated site.
  • Protected docs for any repo. Use public or private repositories, then decide separately whether the docs should be public or password-protected.
  • Client projects. Document implementation details without exposing the client repository.
  • Internal tools. Give teams a lightweight private docs site without rolling out a full portal or running another server.
  • Same-repo documentation. Review documentation changes alongside code changes instead of syncing a second repo.
  • Release-aware docs. Update docs in the same branch or pull request as the feature, fix, or breaking change they describe.
  • Existing MkDocs sites. Optional import when you already have a MkDocs tree — same publish and delivery flow as starting from scratch (details).
  • Many small docs sites. Serve multiple projects under one shared hostname with one Cloudflare-backed platform.
What it is

A serverless publishing layer.

nrdocs handles onboarding, publishing, routing, access policy, artifact storage, and serving. Your documentation can stay in the project repository, while only the generated site is published.

  • Optional — from MkDocs: nrdocs import mkdocs copies structure locally, then nrdocs init adds the same workflow and config as a greenfield project. nrdocs is a full platform, not only a migration helper.
What it is not

Not another docs religion.

nrdocs is not trying to replace MkDocs, Docusaurus, Starlight, or your favorite static site generator. It is the publish, routing, and access layer on top of plain Markdown content in your repo.

Open source

Run it yourself. Change it if you need to.

nrdocs is built for people who want a practical, open-source way to publish project documentation from GitHub repositories. It runs without servers on Cloudflare Workers, D1, and R2, supports password-protected docs today, and is designed to grow toward invite-only access without turning into a platform team cosplay exercise.
