## Purpose

This document specifies the GitHub Actions workflow generated by `nrdocs init` and the behavior expected when it runs.

The GitHub Action is the only MVP build environment. nrdocs infrastructure must not clone repos or run repo-provided build commands on Cloudflare.

## Core responsibilities

The GitHub Action must:

1. Check out the repo.
2. Build or render the docs into static artifacts.
3. Obtain a GitHub OIDC token.
4. Upload artifacts to the nrdocs Worker API.
5. Report whether the uploaded artifacts are currently visible.

The GitHub Action must not:

- approve a repo;
- set effective access mode;
- set passwords;
- create or modify auto-approval rules;
- require long-lived publish secrets;
- fail simply because approval is pending.

## Generated workflow path

`nrdocs init` creates:

```text
.github/workflows/nrdocs.yml
```

## Workflow name

```yaml
name: nrdocs
```

## Trigger

MVP default:

```yaml
on:
  push:
    branches:
      - main
  workflow_dispatch:
```

If the repo default branch is not `main`, `nrdocs init` may detect it and use the current default branch.

The generated workflow should be easy to edit, but the defaults should be safe.

## Required permissions

The workflow must include:

```yaml
permissions:
  contents: read
  id-token: write
```

Rationale:

- `contents: read` allows checkout.
- `id-token: write` allows GitHub OIDC authentication to nrdocs.

The workflow must not require a GitHub secret containing a long-lived nrdocs publish token.

## Minimal generated workflow

Example MVP workflow:

```yaml
name: nrdocs

on:
  push:
    branches:
      - main
  workflow_dispatch:

permissions:
  contents: read
  id-token: write

jobs:
  publish-docs:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Publish docs
        env:
          NRDOCS_API_URL: https://docs.example.com/api
        run: npx nrdocs publish
```

The package manager and runtime may change depending on implementation, but the workflow behavior must remain the same.

## Configurable fields

`nrdocs init` may generate values for:

```text
NRDOCS_API_URL
branch trigger
Node/runtime version
publish command
```

Repo owners may edit the workflow to add custom build preparation steps before `nrdocs publish`, such as dependency installation, but MVP nrdocs itself should not require arbitrary server-side builds.

## Build/render behavior

MVP uses one rendering model only:

```text
Model A: nrdocs CLI renders Markdown into static HTML and uploads the resulting artifact.
```

The workflow calls:

```bash
nrdocs publish
```

The CLI reads `docs/nrdocs.yml`, renders Markdown, copies validated local assets, creates a static artifact archive, and uploads that archive to the Worker API.

Custom output upload, such as the following, is future work and must not be implemented for MVP:

```bash
nrdocs publish --output dist/docs
```

The MVP GitHub Action may run setup steps before `nrdocs publish`, but nrdocs itself owns the Markdown-to-HTML rendering path.

## OIDC authentication

The publish command must request a GitHub OIDC token from GitHub Actions.

The server must verify the token and derive identity from verified claims.

Important claims include:

```text
repository
repository_id
repository_owner
repository_owner_id
ref
sha
workflow_ref
job_workflow_ref
event_name
run_id
run_attempt
```

The authoritative repo identity must come from the token, not from workflow environment variables or request body fields.

## Publish API interaction

The workflow calls the CLI, and the CLI calls the Worker API publish endpoint.

Conceptual request:

```text
POST /api/publish
Authorization: Bearer <github-oidc-token>
Content-Type: multipart/form-data or application/octet-stream
```

The exact upload protocol is defined in `05-api-spec.md` and `09-artifact-storage.md`.

## First publish behavior

If this is the first time a repo publishes:

1. Worker verifies OIDC token.
2. Worker creates repo record.
3. Worker stores uploaded artifact.
4. Worker evaluates auto-approval rules.
5. Worker returns status.

If no auto-approval rule matches:

```text
build: ready
approval: pending
access: none
serving: not_visible
```

The GitHub Action must still succeed.

## Pending approval is not failure

This is mandatory.

The Action should fail on:

- invalid docs config;
- docs rendering/build failure;
- OIDC unavailable;
- OIDC rejected;
- artifact validation failure;
- upload failure;
- API failure.

The Action must succeed on:

- artifact uploaded and repo is pending approval;
- artifact uploaded and repo is approved public;
- artifact uploaded and repo is approved password;
- artifact uploaded and repo is auto-approved password but needs a password credential before serving.

The Action must fail when the API rejects the publish because the repo is disabled. Disabled repos are operator-blocked and must be rejected before artifact storage.

For disabled repos, output should be explicit:

```text
Publish rejected: this repo has been disabled by the operator.
```

## GitHub Step Summary

The workflow should write a GitHub Step Summary when possible.

Pending example:

```markdown
## nrdocs

Docs were uploaded successfully.

| Field | Value |
|---|---|
| Repo | `noam-r/repoA` |
| Build | ready |
| Approval | pending |
| Access | none |
| Serving | not visible |
| URL | https://docs.example.com/noam-r/repoA/ |

An nrdocs operator must approve this repo before the docs are visible.
```

Approved public example:

```markdown
## nrdocs

Docs were uploaded and are publicly available.

| Field | Value |
|---|---|
| Repo | `noam-r/repoA` |
| Build | ready |
| Approval | approved |
| Access | public |
| URL | https://docs.example.com/noam-r/repoA/ |
```

Approved password example:

```markdown
## nrdocs

Docs were uploaded and are available behind password protection.

| Field | Value |
|---|---|
| Repo | `noam-r/repoA` |
| Build | ready |
| Approval | approved |
| Access | password |
| URL | https://docs.example.com/noam-r/repoA/ |
```

## Workflow output rules

The workflow must not print:

- OIDC tokens;
- operator tokens;
- passwords;
- signed upload URLs;
- cookies;
- full artifact contents.

It may print:

- repo full name;
- build ID;
- commit SHA;
- approval status;
- access mode;
- docs URL;
- non-sensitive validation errors.

## Branch behavior

MVP default should publish only from the selected branch, usually the default branch.

The server should record:

```text
ref
sha
```

The server may reject publishes from unexpected refs if deployment policy later supports branch restrictions.

For MVP, branch restriction can be handled by the generated workflow trigger.

## Pull request behavior

MVP should not publish on pull requests by default.

Do not generate:

```yaml
on:
  pull_request:
```

unless preview deployments are explicitly added later.

Reason: publishing PR artifacts from forks introduces additional security and privacy complexity.

## Fork behavior

The MVP must not rely on publishing from forked pull requests.

If a fork runs the workflow manually and attempts to publish, the server should verify the OIDC token and derive the fork repo identity. It must not allow the fork to publish as the upstream repo.

## Manual dispatch behavior

`workflow_dispatch` is allowed so repo owners can re-run publishing without creating a new commit.

This does not violate the invariant that operator approval does not require a new push. Manual dispatch is only for rebuilding or retrying failed uploads.

## Re-run behavior

Re-running a successful workflow may upload an equivalent artifact again.

The server may de-duplicate by content hash, but correctness must not depend on de-duplication.

## Artifact packaging requirements

The CLI should package the output deterministically when possible.

The artifact must not include:

- `.git/`;
- GitHub workflow secrets;
- environment dumps;
- `node_modules/`, unless explicitly part of static output and allowed by artifact policy;
- files outside the configured docs/output directory;
- symlinks resolving outside the output directory.

Detailed artifact rules belong in `09-artifact-storage.md`.

## Status response handling

The publish API returns a response like:

```json
{
  "repo": {
    "full_name": "noam-r/repoA"
  },
  "build": {
    "status": "ready",
    "id": "build_123"
  },
  "approval_state": "pending",
  "access_mode": "none",
  "serving": {
    "status": "not_visible",
    "url": "https://docs.example.com/noam-r/repoA/"
  }
}
```

The Action summary and logs must use this response as the source of truth.

## Error handling

### Invalid config

Action fails.

Example:

```text
nrdocs config error: docs/nrdocs.yml field site.title must be a non-empty string.
```

### Build/render failure

Action fails.

Example:

```text
nrdocs build failed: docs/index.md could not be rendered.
```

### Missing OIDC permission

Action fails with actionable message.

Example:

```text
GitHub OIDC token is unavailable.
Ensure the workflow has:

permissions:
  id-token: write
```

### Pending approval

Action succeeds.

Example:

```text
Docs uploaded successfully. Awaiting operator approval.
```

### Password not configured

If repo is approved with password access but no password credential exists, Action succeeds and reports:

```text
Docs uploaded successfully. Repo is approved for password access, but an operator must configure a password before readers can access it.
```

## Required acceptance tests

1. Generated workflow includes `id-token: write`.
2. Generated workflow does not require `NRDOCS_PUBLISH_TOKEN`.
3. First publish for unknown repo uploads artifact and exits successfully when pending.
4. First publish for auto-approved repo uploads artifact and exits successfully with visible status.
5. Invalid config fails the Action.
6. Missing OIDC permission fails the Action with a clear message.
7. Pull request trigger is not generated by default.
8. Workflow logs do not expose tokens or passwords.

## Non-goals for MVP

The MVP GitHub Action does not provide:

- PR preview deployments;
- comments on pull requests;
- GitHub App installation flow;
- repo-owner OAuth approval requests;
- custom domain provisioning;
- server-side builds;
- cross-repo aggregation;
- publishing from local machines.
