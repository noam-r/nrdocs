# nrdocs Data Model

## Purpose

This document specifies the persistent data model for nrdocs.

The data model must support a serverless, protected-first publishing flow:

```text
Repo owners upload generated docs artifacts.
Operators decide whether and how those artifacts are served.
```

This document is normative. Any implementation must preserve the invariants in [`01-non-negotiable-invariants.md`](./01-non-negotiable-invariants.md).

## Design Principles

### 1. Separate Artifact State from Serving Policy

A repo may have uploaded artifacts while still being invisible.

Therefore, build state and access policy must be modeled separately.

Bad model:

```text
published = true
```

Good model:

```text
latest_successful_build_id = build_123
approval_state = pending
access_mode = none
```

### 2. GitHub Repository ID Is the Stable Identity

GitHub `repository_id` is the primary stable identity.

`OWNER/REPO` is a display and routing identity.

This matters because repositories can be renamed.

### 3. Operator Policy Is Authoritative

Repo config may request access preferences, but the database stores the effective operator-controlled policy.

The serving Worker must use D1 policy state, not repo config, to decide visibility.

### 4. Approval and Access Are Metadata

Approving a repo or changing access mode must only update metadata.

It must not require:

```text
- New GitHub push
- New artifact upload
- Artifact copy
- Artifact rewrite
- Static file challenge generation
```

## Logical Entities

The MVP data model contains these entities:

```text
repos
builds
password_credentials
auto_approval_rules
audit_log
```

Optional later entities:

```text
custom_routes
reader_sessions
repo_aliases
operator_tokens
```

Reader sessions may also be implemented as signed cookies without a D1 table.

## Entity: repos

The `repos` table represents a GitHub repository that has interacted with nrdocs through a verified GitHub OIDC publish. Operators must not create fake approved repo rows before OIDC establishes the immutable GitHub repository ID. Pre-approval is represented with `auto_approval_rules`, not unverified `repos` rows.

### Required Fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| `id` | string | yes | Internal repo ID, generated by nrdocs |
| `github_repository_id` | string | yes | Stable GitHub repository ID from OIDC claims |
| `owner` | string | yes | Current GitHub owner/login |
| `name` | string | yes | Current GitHub repository name |
| `full_name` | string | yes | Current `OWNER/REPO`, stored lowercase |
| `default_branch` | string | no | Default branch when known |
| `approval_state` | enum | yes | `pending`, `approved`, or `disabled` |
| `access_mode` | enum | yes | `none`, `public`, or `password` |
| `latest_successful_build_id` | string | no | Build currently selected for serving |
| `last_publish_status` | enum | no | Last publish outcome summary |
| `requested_access` | enum | no | Advisory value from repo config |
| `site_title` | string | no | Display title from repo config |
| `created_at` | timestamp | yes | First seen time |
| `updated_at` | timestamp | yes | Last metadata update |
| `approved_at` | timestamp | no | Last approval time |
| `approved_by` | string | no | Operator or rule ID that approved repo |
| `disabled_at` | timestamp | no | Last disabled time |
| `disabled_by` | string | no | Operator that disabled repo |

### Required Constraints

```text
github_repository_id must be unique.
full_name must be stored lowercase and indexed for lookup and routing.
owner and name must be stored lowercase.
Route matching must normalize incoming paths to lowercase before lookup.
latest_successful_build_id must refer to a build belonging to the same repo.
```

### Approval State

Allowed values:

| Value | Meaning | Anonymous serving allowed? |
|---|---|---:|
| `pending` | Repo has not been approved | no |
| `approved` | Repo may be served according to access mode | maybe |
| `disabled` | Repo is explicitly blocked | no |

### Access Mode

Allowed values:

| Value | Meaning |
|---|---|
| `none` | No serving access is allowed |
| `public` | Approved repo may be served publicly |
| `password` | Approved repo requires password/session |

### Valid Serving Combinations

| approval_state | access_mode | Serve? |
|---|---|---:|
| `pending` | `none` | no |
| `pending` | `public` | no; should not normally occur |
| `pending` | `password` | no; should not normally occur |
| `approved` | `none` | no |
| `approved` | `public` | yes, if build exists |
| `approved` | `password` | yes, after password/session validation, build exists, and a password credential exists; otherwise serving substatus is `needs_password` and anonymous readers do not get docs |
| `disabled` | any | no |

Implementation should prefer preventing invalid combinations, but the serving Worker must still enforce denial safely if invalid data exists.

Serving substatus `needs_password` is derived, not a separate approval state. It applies when `approval_state = approved`, `access_mode = password`, a successful build exists, and no password credential exists.

### Default Values for New Repos

When a repo publishes for the first time and no auto-approval rule matches:

```text
approval_state = pending
access_mode = none
latest_successful_build_id = newly uploaded build, if upload succeeded
```

This is the protected-first default.

## Entity: builds

The `builds` table represents a generated docs artifact upload.

### Required Fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| `id` | string | yes | Internal build ID |
| `repo_id` | string | yes | Owning repo |
| `github_repository_id` | string | yes | Duplicated for audit/debugging |
| `git_sha` | string | yes | Commit SHA that produced the build |
| `git_ref` | string | no | Git ref, such as `refs/heads/main` |
| `workflow_ref` | string | no | GitHub workflow ref from OIDC claims |
| `run_id` | string | no | GitHub Actions run ID |
| `status` | enum | yes | `uploading`, `success`, `failed` |
| `artifact_prefix` | string | no | R2 prefix for this build |
| `artifact_manifest_key` | string | no | R2 key for manifest file, if used |
| `artifact_size_bytes` | integer | no | Total uploaded artifact size |
| `file_count` | integer | no | Number of files uploaded |
| `content_hash` | string | no | Digest of artifact bundle or manifest |
| `error_code` | string | no | Failure code |
| `error_message` | string | no | Sanitized failure message |
| `created_at` | timestamp | yes | Build record creation time |
| `completed_at` | timestamp | no | Upload completion time |

### Build Status

Allowed values:

| Value | Meaning |
|---|---|
| `uploading` | Build record exists but artifact is not complete |
| `success` | Artifact upload completed and may be selected for serving |
| `failed` | Build failed or upload did not complete |

Only `success` builds may become `repos.latest_successful_build_id`.

### Build Commit Rule

A build must not become the latest successful build until:

```text
1. All artifact files are uploaded.
2. Manifest is stored, if manifest is used.
3. Build status is set to success.
4. Repo latest_successful_build_id is updated atomically or in a safe ordered sequence.
```

A partial upload must never be served.

## Entity: password_credentials

The `password_credentials` table stores password verification data for repos using password access.

### Required Fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| `id` | string | yes | Internal credential ID |
| `repo_id` | string | yes | Repo this password protects |
| `password_hash` | string | yes | Password hash, never plaintext |
| `hash_algorithm` | string | yes | Algorithm identifier (e.g., `pbkdf2-sha256`) |
| `salt` | string | yes | Per-password random salt |
| `iteration_count` | integer | yes | PBKDF2 iteration count used for this hash |
| `password_version` | integer | yes | Incremented on each password change; used for session invalidation |
| `created_at` | timestamp | yes | Creation time |
| `updated_at` | timestamp | yes | Last change time |
| `updated_by` | string | yes | Operator identity |
| `active` | boolean | yes | Whether credential is active |

### Password Rules

1. Plaintext passwords must never be stored.
2. Plaintext passwords must never be written to audit logs.
3. Password artifacts must not be generated into the static docs output.
4. Password changes must not require a repo rebuild.

For MVP, one active password per repo is sufficient.

## Entity: auto_approval_rules

The `auto_approval_rules` table stores operator-defined rules that approve matching repos automatically.

### Required Fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| `id` | string | yes | Internal rule ID |
| `pattern` | string | yes | Match pattern such as `noam-r/*` |
| `access_mode` | enum | yes | `public` or `password` |
| `enabled` | boolean | yes | Whether rule is active |
| `priority` | integer | yes | Rule ordering when multiple rules match |
| `created_at` | timestamp | yes | Creation time |
| `created_by` | string | yes | Operator identity |
| `updated_at` | timestamp | yes | Last change time |
| `updated_by` | string | yes | Operator identity |

### Pattern Syntax

MVP supports two pattern forms:

```text
OWNER/*
OWNER/REPO
```

Examples:

```text
noam-r/*
noam-r/repoA
my-company/internal-docs
```

Patterns must be stored and matched in lowercase. Matching normalizes both the pattern and the incoming repo identity to lowercase before comparison.

No other glob syntax is required for MVP.

### Rule Semantics

A matching rule may set:

```text
approval_state = approved
access_mode = rule.access_mode
approved_by = auto_rule:<rule_id>
```

Auto-approval rules must not grant repo owners control over access mode, custom slugs, domains, or headers.

### Rule Matching Order

If multiple rules match, use this deterministic order:

```text
1. Higher priority wins.
2. Exact OWNER/REPO match wins over OWNER/* when priorities are equal.
3. Older rule wins when still tied.
```

The selected rule must be recorded in the audit log.

## Operator Token Storage

MVP operator authentication uses exactly one deployment-level operator token configured as a Cloudflare Worker secret. The token is not represented by a D1 table in MVP.

Required behavior:

1. Store the operator token only as a Worker secret or equivalent deployment secret.
2. Do not store plaintext operator tokens in D1, repo config, generated workflows, audit logs, or static files.
3. Compare supplied operator tokens using a constant-time comparison or equivalent safe comparison.
4. Multi-token management, token metadata tables, token scopes, and revocation records are future work.

Future implementations may add an `operator_tokens` table, but it is not part of the MVP schema.

## Entity: audit_log

The `audit_log` table records security-relevant events.

### Required Fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| `id` | string | yes | Internal event ID |
| `event_type` | string | yes | Event name |
| `actor_type` | enum | yes | `operator`, `github_action`, `system` |
| `actor_id` | string | no | Operator ID, repo ID, or rule ID |
| `repo_id` | string | no | Related repo |
| `build_id` | string | no | Related build |
| `rule_id` | string | no | Related auto-approval rule |
| `metadata_json` | string | no | Sanitized event metadata |
| `created_at` | timestamp | yes | Event time |

### Required Audit Events

```text
repo.first_seen
repo.published
repo.auto_approved
repo.manually_approved
repo.disabled
repo.access_changed
repo.password_set
repo.password_rotated
auto_rule.created
auto_rule.updated
auto_rule.deleted
operator.auth_failed
publish.oidc_failed
publish.failed
```

Audit metadata must not include plaintext passwords, raw operator tokens, or artifact contents.

## Optional Entity: repo_aliases

Repo aliases may be useful after GitHub repo renames.

MVP can skip this table, but the implementation should still treat `github_repository_id` as stable.

Suggested fields:

| Field | Type | Description |
|---|---:|---|
| `id` | string | Internal alias ID |
| `repo_id` | string | Current repo |
| `old_full_name` | string | Previous `OWNER/REPO` |
| `new_full_name` | string | New `OWNER/REPO` |
| `created_at` | timestamp | Alias creation time |

For MVP, a rename may simply update `owner`, `name`, and `full_name` when the same `github_repository_id` publishes again.

A transfer to a different owner should be treated conservatively according to the security spec.

## State Transitions

### First Publish, No Matching Auto-Approval Rule

```text
No repo record
  -> create repo
  -> approval_state = pending
  -> access_mode = none
  -> create successful build
  -> latest_successful_build_id = build_id
```

### First Publish, Matching Auto-Approval Rule

```text
No repo record
  -> create repo
  -> create successful build
  -> latest_successful_build_id = build_id
  -> approval_state = approved
  -> access_mode = matching_rule.access_mode
```

### Manual Approval

```text
approval_state = pending
access_mode = none
  -> approval_state = approved
  -> access_mode = operator-selected public|password
```

### Disable Repo

```text
any state
  -> approval_state = disabled
  -> access_mode may remain unchanged or be set to none
```

Serving must deny disabled repos regardless of access mode.

### Later Successful Publish

```text
existing repo
  -> create new build
  -> upload artifacts
  -> build.status = success
  -> latest_successful_build_id = new_build_id
  -> approval_state unchanged, unless pending and an auto-approval rule now matches
  -> access_mode unchanged, unless pending and an auto-approval rule now matches
```

### Failed Publish

```text
existing repo
  -> create failed build or mark current build failed
  -> latest_successful_build_id unchanged
```

A failed publish must not break the currently served version.

## Serving Decision Algorithm

The serving Worker must use logic equivalent to:

```text
repo = findRepoByRoute(owner, repo)

if repo is missing:
  return 404

if repo.approval_state != approved:
  return 404

if repo.latest_successful_build_id is missing:
  return 404 or unavailable page

if repo.access_mode == public:
  serve artifact

if repo.access_mode == password:
  if valid password session:
    serve artifact
  else:
    return password page

return 404
```

Invalid or unexpected combinations must fail closed.

## Suggested SQL Shape

The exact SQL may differ, but this captures the intended schema.

```sql
CREATE TABLE repos (
  id TEXT PRIMARY KEY,
  github_repository_id TEXT NOT NULL UNIQUE,
  owner TEXT NOT NULL,
  name TEXT NOT NULL,
  full_name TEXT NOT NULL,
  default_branch TEXT,
  approval_state TEXT NOT NULL CHECK (approval_state IN ('pending', 'approved', 'disabled')),
  access_mode TEXT NOT NULL CHECK (access_mode IN ('none', 'public', 'password')),
  latest_successful_build_id TEXT,
  last_publish_status TEXT,
  requested_access TEXT,
  site_title TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  approved_at TEXT,
  approved_by TEXT,
  disabled_at TEXT,
  disabled_by TEXT
);

CREATE INDEX idx_repos_full_name ON repos(full_name);
CREATE INDEX idx_repos_approval_state ON repos(approval_state);
```

```sql
CREATE TABLE builds (
  id TEXT PRIMARY KEY,
  repo_id TEXT NOT NULL,
  github_repository_id TEXT NOT NULL,
  git_sha TEXT NOT NULL,
  git_ref TEXT,
  workflow_ref TEXT,
  run_id TEXT,
  status TEXT NOT NULL CHECK (status IN ('uploading', 'success', 'failed')),
  artifact_prefix TEXT,
  artifact_manifest_key TEXT,
  artifact_size_bytes INTEGER,
  file_count INTEGER,
  content_hash TEXT,
  error_code TEXT,
  error_message TEXT,
  created_at TEXT NOT NULL,
  completed_at TEXT,
  FOREIGN KEY (repo_id) REFERENCES repos(id)
);

CREATE INDEX idx_builds_repo_id ON builds(repo_id);
CREATE INDEX idx_builds_status ON builds(status);
```

```sql
CREATE TABLE password_credentials (
  id TEXT PRIMARY KEY,
  repo_id TEXT NOT NULL,
  password_hash TEXT NOT NULL,
  hash_algorithm TEXT NOT NULL,
  salt TEXT NOT NULL,
  iteration_count INTEGER NOT NULL,
  password_version INTEGER NOT NULL DEFAULT 1,
  active INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  updated_by TEXT NOT NULL,
  FOREIGN KEY (repo_id) REFERENCES repos(id)
);

CREATE INDEX idx_password_credentials_repo_id ON password_credentials(repo_id);
```

```sql
CREATE TABLE auto_approval_rules (
  id TEXT PRIMARY KEY,
  pattern TEXT NOT NULL,
  access_mode TEXT NOT NULL CHECK (access_mode IN ('public', 'password')),
  enabled INTEGER NOT NULL DEFAULT 1,
  priority INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  created_by TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  updated_by TEXT NOT NULL
);

CREATE INDEX idx_auto_approval_rules_enabled ON auto_approval_rules(enabled);
```

```sql
CREATE TABLE audit_log (
  id TEXT PRIMARY KEY,
  event_type TEXT NOT NULL,
  actor_type TEXT NOT NULL CHECK (actor_type IN ('operator', 'github_action', 'system')),
  actor_id TEXT,
  repo_id TEXT,
  build_id TEXT,
  rule_id TEXT,
  metadata_json TEXT,
  created_at TEXT NOT NULL
);

CREATE INDEX idx_audit_log_repo_id ON audit_log(repo_id);
CREATE INDEX idx_audit_log_created_at ON audit_log(created_at);
```

## Data Model Invariants

The following invariants must be enforced in application logic, database constraints, or both:

1. A repo is not serveable unless `approval_state = approved`.
2. A repo is not publicly serveable unless `approval_state = approved` and `access_mode = public`.
3. A password-protected repo is not serveable unless `approval_state = approved`, `access_mode = password`, and the reader has a valid password session.
4. `latest_successful_build_id` must only reference a successful build.
5. Failed or uploading builds must not be served.
6. Auto-approval rules must specify `public` or `password`; there is no implicit public default.
7. New repos default to `pending` and `none` unless an enabled auto-approval rule matches.
8. Operator access changes must not modify artifact contents.
9. Plaintext passwords must never be stored.
10. Unknown, pending, disabled, and unapproved repos must fail closed.

## Acceptance Criteria

An implementation satisfies this data model when:

1. First publish creates a protected, non-visible repo by default.
2. Artifacts can be stored for pending repos.
3. Manual approval can make existing artifacts visible without another publish.
4. Public access is impossible without operator policy.
5. Auto-approval can approve matching repos with explicit access mode.
6. Disabling a repo immediately prevents serving.
7. Later successful publishes replace the served artifact only after complete upload.
8. Failed publishes do not affect the last successful served build.
9. Serving decisions can be made entirely from D1 metadata and R2 artifact presence.
10. The data model supports all user flows in [`02-user-flows.md`](./02-user-flows.md).
