# Requirements Document

## Introduction

The nrdocs system currently requires the operator (the person who deployed the nrdocs Worker) to set the password for every password-protected docs site. This creates an operational bottleneck and forces repo owners to share a secret with the operator out-of-band.

This feature lets the operator opt a repo in to self-service password management. When opted in, the repo owner stores a password in a GitHub Actions Secret, the publish workflow forwards it to the Worker over TLS as part of the existing publish request, and the Worker hashes and stores it on the repo record. Repos that are not opted in continue to behave as today: any password sent during publish is silently ignored so that a shared workflow template does not leak which repos are gated.

The change extends the existing publish allowlist model. Operators retain full authority: they decide which repos may self-serve, they remain the sole authority for changing access mode away from `password`, and they keep the existing operator-only `nrdocs password set` command as a fallback.

## Glossary

- **Worker**: The deployed nrdocs Cloudflare Worker that authenticates publish requests and stores repo, build, and password records in D1.
- **Operator**: The user who deployed the Worker and holds `OPERATOR_TOKEN`. Authenticates to the Worker via Bearer token from the `nrdocs` CLI.
- **Repo_Owner**: A GitHub user or organization that publishes docs from one of their repos to the operator's nrdocs instance via GitHub Actions OIDC.
- **CLI**: The `nrdocs` Node.js CLI distributed on npm, used by both operators and repo owners.
- **Publish_Request**: A `POST /api/publish` multipart request from a GitHub Actions runner authenticated with a GitHub OIDC token bound to audience `nrdocs`.
- **Publish_Allowlist**: The existing rule that rejects publish requests unless the repo already exists in the `repos` table OR matches an enabled auto-approval rule.
- **Allow_Self_Password_Flag**: A new boolean column `allow_repo_owner_password` on the `repos` table, default `false`, set only by an operator.
- **Self_Service_Password**: A password supplied by a repo owner via the publish flow rather than by an operator via `nrdocs password set`.
- **Password_Field**: The multipart form field named `password` in a Publish_Request, containing the plaintext password supplied by the repo owner.
- **Workflow_Secret**: A GitHub Actions encrypted Secret named `NRDOCS_DOCS_PASSWORD` configured by the Repo_Owner on their repository.
- **Workflow_Template**: The `.github/workflows/nrdocs.yml` file generated by `nrdocs init` and updated when repo owners re-run init.
- **Operator_CLI_Command**: A subcommand of the `nrdocs` CLI that requires `OPERATOR_TOKEN` credentials.
- **Hash_Pipeline**: The existing PBKDF2-HMAC-SHA-256 pipeline (100,000 iterations, 16-byte salt, 32-byte derived key) used by `setPassword()` in the Worker.
- **Migration**: A SQL file under `packages/worker/migrations/` applied to the D1 database in lexicographic filename order.
- **Rule_Self_Password_Default**: A boolean column `default_allow_repo_owner_password` on `auto_approval_rules`, default `true`, that determines the value of `allow_repo_owner_password` stamped onto a brand-new repo row at the moment a Publish_Request from that repo is auto-approved by the rule. The default is set at rule creation time via the `--self-set-password allow|deny` CLI flag (defaulting to `allow` when the flag is omitted).

## Requirements

### Requirement 1: Per-repo opt-in flag

**User Story:** As an operator, I want to explicitly mark which repos may self-set their docs password, so that no repo can change its own password without my consent.

#### Acceptance Criteria

1. THE Worker SHALL persist a boolean field `allow_repo_owner_password` on every repo record, with default value `false` for both new and pre-existing repos.
2. WHEN the Worker reads a repo record, THE Worker SHALL include `allow_repo_owner_password` in the returned `RepoRecord` shape used by all internal callers.
3. WHEN the Migration is applied to a database that already contains repo rows, THE Migration SHALL set `allow_repo_owner_password` to `false` for every existing row without modifying any other column, AND existing operator-set passwords stored in `password_credentials` SHALL remain valid and active so that operator-driven password management continues to work unchanged after the migration.
4. THE Worker SHALL validate the `OPERATOR_TOKEN` Bearer credential before performing any state read or mutation related to `allow_repo_owner_password`, AND IF the credential is missing or invalid, THEN THE Worker SHALL respond immediately with HTTP status 401 without modifying any record.
5. IF the Migration cannot be applied (for example because the column already exists), THEN THE Migration SHALL fail with a non-zero exit status and SHALL NOT partially modify the schema.

### Requirement 2: Operator allow/disallow CLI commands

**User Story:** As an operator, I want simple CLI subcommands to toggle self-service password permission for a specific repo, so that I can grant or revoke the capability without writing SQL.

#### Acceptance Criteria

1. WHEN the operator runs `nrdocs password allow OWNER/REPO` against an existing repo, THE Worker SHALL set `allow_repo_owner_password` to `true` on that repo and THE CLI SHALL print a success message naming the repo.
2. WHEN the operator runs `nrdocs password disallow OWNER/REPO` against an existing repo, THE Worker SHALL set `allow_repo_owner_password` to `false` on that repo and THE CLI SHALL print a success message naming the repo.
3. THE CLI SHALL expose `allow` and `disallow` strictly as subcommands of the existing `nrdocs password` command and SHALL NOT introduce top-level commands named `allow-self-password` or `disallow-self-password`.
4. THE existing `nrdocs password set OWNER/REPO` operator override SHALL continue to function unchanged when the new subcommands are added.
5. IF the target repo does not exist in the `repos` table, THEN THE Worker SHALL respond with HTTP status 404 and error code `NOT_FOUND`, and THE CLI SHALL exit with a non-zero exit status and a human-readable error message.
6. IF the CLI is invoked with a repository argument that is not in `OWNER/REPO` format, THEN THE CLI SHALL exit with status 2 and print a usage message.
7. IF the CLI is invoked without resolvable operator credentials, THEN THE CLI SHALL exit with status 1 and print an error directing the user to `nrdocs auth login`.
8. WHEN either subcommand succeeds, THE Worker SHALL write an entry to `audit_log` with `actor_type='operator'`, `repo_id` set to the affected repo, and `event_type` of `repo.self_password_allowed` (for allow) or `repo.self_password_disallowed` (for disallow).
9. WHEN either subcommand is run twice in a row with the same value, THE Worker SHALL succeed both times (idempotent) and SHALL still write an audit entry on each call.

### Requirement 3: Workflow template injects the secret

**User Story:** As a repo owner, I want `nrdocs init` to wire `NRDOCS_DOCS_PASSWORD` into my publish workflow automatically, so that I do not have to hand-edit YAML to use self-service passwords.

#### Acceptance Criteria

1. WHEN `nrdocs init` writes the Workflow_Template, THE CLI SHALL include the line `NRDOCS_DOCS_PASSWORD: ${{ secrets.NRDOCS_DOCS_PASSWORD }}` in the `env:` block of the publish step.
2. THE Workflow_Template SHALL place `NRDOCS_DOCS_PASSWORD` alongside the existing `NRDOCS_API_URL` env entry without removing or reordering any pre-existing required entries.
3. WHERE the Repo_Owner has not configured the Workflow_Secret, THE Workflow_Template SHALL still execute successfully and `NRDOCS_DOCS_PASSWORD` SHALL be the empty string at runtime.
4. WHEN `nrdocs init` is run with `--force` against a repository that already has a Workflow_Template, THE CLI SHALL overwrite the workflow file and the new file SHALL contain the `NRDOCS_DOCS_PASSWORD` env line.

### Requirement 4: CLI publish forwards the password

**User Story:** As a repo owner, I want the publish CLI to send my secret password to the Worker only when I have configured one, so that I do not accidentally send empty or stale credentials.

#### Acceptance Criteria

1. WHEN the CLI builds the publish multipart body and `process.env.NRDOCS_DOCS_PASSWORD` is set to a non-empty string (any string of length >= 1, including whitespace-only strings), THE CLI SHALL append a multipart field named `password` whose value is the environment variable contents.
2. WHEN `process.env.NRDOCS_DOCS_PASSWORD` is unset or is the empty string (length 0), THE CLI SHALL omit the `password` field from the multipart body entirely.
3. THE CLI SHALL transmit the publish request over HTTPS to the configured `api_url` (the same channel used today for OIDC-authenticated publish requests).
4. THE CLI SHALL NOT log the value of `NRDOCS_DOCS_PASSWORD` at any verbosity level, and SHALL NOT write the value to stdout, stderr, archive contents, or any local file.
5. IF the publish request fails for any reason after the Password_Field is appended, THEN THE CLI SHALL produce its existing error output without including the password value.

### Requirement 5: Worker honors the allow flag during publish

**User Story:** As an operator, I want the Worker to accept self-service passwords only for repos I have opted in, so that the allow flag is the single source of truth for who can self-set passwords.

#### Acceptance Criteria

1. WHEN a Publish_Request is processed and a `password` multipart field is present, THE Worker SHALL extract the field value before evaluating the Allow_Self_Password_Flag.
2. WHILE `allow_repo_owner_password` is `true` for the target repo AND the extracted password value is at least `DEFAULT_MIN_PASSWORD_LENGTH` characters and at most `DEFAULT_MAX_PASSWORD_LENGTH` characters, THE Worker SHALL hash the password through the Hash_Pipeline and persist the result via the existing `setPassword` helper with `updated_by='repo_owner'`.
3. WHILE `allow_repo_owner_password` is `false` for the target repo, THE Worker SHALL discard the extracted password value without hashing or storing it, and SHALL continue processing the publish request as if the field had not been sent.
4. IF the password field is present but its length is outside the configured min/max bounds AND `allow_repo_owner_password` is `true`, THEN THE Worker SHALL discard the password without storing it and SHALL respond with HTTP 400 and error code `INVALID_PASSWORD` describing only that the supplied password failed length validation.
5. WHEN the Worker stores a self-service password, THE Worker SHALL write an `audit_log` entry with `event_type='repo.self_password_set'`, `actor_type='github_action'`, `actor_id=<full_name>`, and `build_id` of the in-flight build.
6. WHEN the Worker discards a self-service password because the flag is `false`, THE Worker SHALL write an `audit_log` entry with `event_type='repo.self_password_ignored'`, `actor_type='github_action'`, `actor_id=<full_name>`, and SHALL NOT include the plaintext or any hash of the password in the metadata.
7. THE Worker SHALL NOT include the values `repo.self_password_set` or `repo.self_password_ignored` (or any password-related discriminant) in the publish response body, so that the response is byte-for-byte indistinguishable to the Repo_Owner regardless of the flag state.
8. THE Worker response SHALL NOT contain the plaintext password value, the derived hash, or the salt under any field name.
9. IF the audit log write for a self-service password set or ignore event cannot complete successfully, THEN THE Worker SHALL roll back or otherwise abort the password storage and SHALL respond with HTTP 500 and error code `AUDIT_WRITE_FAILED`.

### Requirement 6: Access-mode interaction during self-service password set

**User Story:** As an operator, I want clear and predictable rules for how a self-service password affects the repo's access mode, so that a repo owner cannot bypass an operator-set access policy by uploading a password.

#### Acceptance Criteria

1. WHILE the repo is in `approval_state='approved'` AND `access_mode='password'` AND `allow_repo_owner_password=true`, WHEN a valid self-service password is supplied, THE Worker SHALL store the new password and SHALL leave `access_mode` unchanged.
2. WHILE the repo is in `approval_state='approved'` AND `access_mode='none'` AND `allow_repo_owner_password=true`, WHEN a valid self-service password is supplied, THE Worker SHALL store the new password and SHALL set `access_mode` to `password`.
3. WHILE the repo is in `approval_state='approved'` AND `access_mode='public'` AND `allow_repo_owner_password=true`, WHEN a valid self-service password is supplied, THE Worker SHALL store the new password and SHALL leave `access_mode` set to `public` (the password becomes available for the operator to activate via `nrdocs access set`, but does not automatically gate the public site).
4. WHILE the repo is in `approval_state='pending'`, WHEN a valid self-service password is supplied AND `allow_repo_owner_password=true`, THE Worker SHALL store the password and SHALL NOT change any other field on the repo record (the existing approval flow continues to apply).
5. WHEN the Worker changes `access_mode` as a side effect of a self-service password set, THE Worker SHALL write an `audit_log` entry with `event_type='repo.access_changed'`, `actor_type='github_action'`, and metadata containing `old_mode` and `new_mode`.
6. IF the Worker would otherwise change `access_mode` but `approval_state` is `disabled`, THEN THE Worker SHALL not update `access_mode` and SHALL not store the password (the existing `REPO_DISABLED` short-circuit takes precedence).

### Requirement 7: Repo owner self-service does not alter publish allowlist behavior

**User Story:** As an operator, I want the existing publish allowlist (known repo or matching auto-approval rule) to remain the gatekeeper for any publish, so that this feature does not widen who can publish at all.

#### Acceptance Criteria

1. THE Worker SHALL evaluate the Publish_Allowlist before extracting or considering the Password_Field.
2. IF a Publish_Request is rejected by the Publish_Allowlist, THEN THE Worker SHALL respond with the existing `REPO_NOT_ALLOWED` error and SHALL NOT read or hash any Password_Field present in the request body.
3. IF a Publish_Request comes from a repo whose `approval_state` is `disabled`, THEN THE Worker SHALL respond with the existing `REPO_DISABLED` error and SHALL NOT read or hash any Password_Field present in the request body.

### Requirement 8: README documents the flow

**User Story:** As a repo owner or operator reading the README, I want a description of the self-service password flow, so that I know how to opt in, where to put the secret, and what behavior to expect.

#### Acceptance Criteria

1. THE README SHALL include a section that explains the operator-side opt-in subcommands `nrdocs password allow` and `nrdocs password disallow` with example invocations.
2. THE README SHALL include instructions for repo owners to create the GitHub Actions Secret `NRDOCS_DOCS_PASSWORD` and a note that it is consumed automatically by the workflow generated by `nrdocs init`.
3. THE README SHALL describe the access-mode interaction rules from Requirement 6 in plain language.
4. THE README SHALL state that supplying `NRDOCS_DOCS_PASSWORD` from a non-opted-in repo has no effect and produces no error.
5. THE README SHALL note that the operator's existing `nrdocs password set` command continues to work as a fallback and as the override path.
6. THE README SHALL document the `--self-set-password allow|deny` flag on `nrdocs rules add`, state that the default when the flag is omitted is `allow`, and explain that the flag controls only the self-password capability stamped onto NEW repos auto-approved by the rule (it does NOT modify already-existing repo rows, and is independent of `--apply-existing` which continues to control only `access_mode` propagation).

### Requirement 9: Test coverage for the new behavior

**User Story:** As a maintainer, I want automated coverage of every new branch in the publish handler and the new CLI commands, so that future refactors do not silently regress the security-sensitive behavior.

#### Acceptance Criteria

1. THE Worker test suite SHALL include tests covering the four states of the matrix `{password field present, password field absent} × {allow flag true, allow flag false}` against the publish endpoint.
2. THE Worker test suite SHALL include a test that verifies the Worker response body is identical for the same publish request whether `allow_repo_owner_password` is `true` (with no password supplied) or `false` (with no password supplied).
3. THE Worker test suite SHALL include a test that verifies a self-service password set with valid credentials results in a stored hash that round-trips via `verifyPassword` for that plaintext.
4. THE CLI test suite SHALL include tests for the `nrdocs password allow` and `nrdocs password disallow` subcommands covering the success path, the missing-credentials path, and the malformed-argument path.
5. THE CLI test suite SHALL include a test verifying that `nrdocs publish` appends a `password` multipart field exactly when `NRDOCS_DOCS_PASSWORD` is set to a non-empty string in the environment.
6. THE CLI test suite SHALL include a test verifying that no command logs the value of `NRDOCS_DOCS_PASSWORD` to stdout or stderr at any verbosity level.
7. THE Worker test suite SHALL include a test verifying that auto-approval of a new repo via a rule with `default_allow_repo_owner_password=true` stamps `allow_repo_owner_password=true` on the new repo row at the moment of the auto-approval insert.
8. THE Worker test suite SHALL include a test verifying that auto-approval of a new repo via a rule with `default_allow_repo_owner_password=false` stamps `allow_repo_owner_password=false` on the new repo row.
9. THE Worker test suite SHALL include a test verifying that adding or updating a rule does not change `allow_repo_owner_password` on any already-existing repo row, regardless of whether the rule's pattern matches those rows and regardless of whether `--apply-existing` was supplied.
10. THE CLI test suite SHALL include parser tests for `nrdocs rules add` covering the cases `--self-set-password allow`, `--self-set-password deny`, omitted (default), and an invalid value (verifying exit status 2 and a usage message).

### Requirement 10: Version bump

**User Story:** As an operator pulling a new CLI release, I want a clear version increment when this feature ships, so that I can pin to a known-good version that includes self-service password support.

#### Acceptance Criteria

1. WHEN this feature's CLI changes are published to npm (which MAY occur after the implementation is merged, at the maintainer's discretion), THE CLI SHALL be published from `packages/cli/package.json` with a version greater than the previous published version.
2. THE CLI version printed by `nrdocs --version` SHALL match the version in `packages/cli/package.json`.
### Requirement 11: Rule-driven self-service password default

**User Story:** As an operator, I want each auto-approval rule to carry a default `allow_repo_owner_password` value so that adding an organization to the allowlist also grants self-service password capability by default, while still letting me deny it explicitly per rule.

#### Acceptance Criteria

1. WHEN the operator runs `nrdocs rules add 'pattern' --access ...` without the `--self-set-password` flag, THE rule SHALL be persisted with `default_allow_repo_owner_password=true`.
2. WHEN the operator runs `nrdocs rules add` with `--self-set-password allow`, THE rule SHALL be persisted with `default_allow_repo_owner_password=true`.
3. WHEN the operator runs `nrdocs rules add` with `--self-set-password deny`, THE rule SHALL be persisted with `default_allow_repo_owner_password=false`.
4. IF the value supplied to `--self-set-password` is anything other than `allow` or `deny`, THEN THE CLI SHALL exit with status 2 and print a usage message naming the valid values, AND SHALL NOT issue any HTTP request to the Worker.
5. WHEN a Publish_Request from a not-yet-approved repo (no row in `repos` for its `github_repository_id`) matches an enabled rule R during the auto-approval insert path, THE Worker SHALL set the new repo row's `allow_repo_owner_password` to `R.default_allow_repo_owner_password` at the moment that row is inserted.
6. THE Worker SHALL NOT retroactively change `allow_repo_owner_password` on any already-existing repo row when a rule is created, updated, or removed, AND `--apply-existing` SHALL continue to control only `access_mode` propagation and SHALL NOT be extended to touch `allow_repo_owner_password`.
7. THE `POST /api/auto-approval-rules` request body SHALL accept an optional boolean `default_allow_repo_owner_password` field, AND THE Worker SHALL default the field to `true` when it is omitted from the body.
8. IF the `default_allow_repo_owner_password` field is present in the request body but is not a boolean, THEN THE Worker SHALL respond with HTTP 400 and a `VALIDATION_ERROR` code naming the field.
9. THE `GET /api/auto-approval-rules` response SHALL include `default_allow_repo_owner_password` for every rule in the returned list.
10. THE `nrdocs rules list` table output SHALL include a column with header `SELF-PWD` whose value is `allow` for rules with `default_allow_repo_owner_password=true` and `deny` for rules with `default_allow_repo_owner_password=false`.
11. WHEN the second migration `0003_rule_self_password_default.sql` is applied to a database that already contains rules, THE Migration SHALL set `default_allow_repo_owner_password` to `1` (allow) for every existing rule without modifying any other column on `auto_approval_rules`.
12. IF the `0003` migration cannot be applied (for example because the column already exists), THEN THE Migration SHALL fail with a non-zero exit status and SHALL NOT partially modify the schema.
